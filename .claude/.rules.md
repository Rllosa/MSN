# Project Rules — Generic Template

These rules govern all development on this project.
They are binding for every task, branch, commit, PR, and architectural decision.
No code may be written, committed, pushed, or merged without following this document.

---

## 0. Operating Mode (Plan-First Discipline)

### 0.1 Plan Before Code

No code changes are allowed until:

- The problem is clearly defined
- Tradeoffs are explained
- A recommended option is proposed
- Explicit user approval is given

### 0.2 Review Mode Selection (Must Ask First)

Before reviewing code or a plan, always ask whether to use:

**1) BIG CHANGE**
Review interactively section-by-section: Architecture → Code Quality → Tests → Performance.
Max 4 top issues per section.

**2) SMALL CHANGE**
Ask one key question per section.

No implementation begins until review decisions are approved.

### 0.3 Engineering Preferences (Guide All Recommendations)

- DRY is important — flag repetition aggressively.
- Tests are non-negotiable — prefer too many tests over too few.
- Code must be "engineered enough" — not fragile, not prematurely abstracted.
- Err on the side of handling more edge cases.
- Prefer explicit over clever.

## 1. Architecture Invariants (Blocking If Violated)

These are hard constraints. Replace these with project-specific invariants before starting.

### 1.1 Deterministic Core

- Core business logic must be pure rules + math where possible.
- No randomness in critical paths. No hidden state.
- Same inputs → same outputs.
- All thresholds, weights, and conditions come from config files, not hardcoded values.
- Zero hardcoded threshold-like numeric values in application code.

### 1.2 External Service Isolation

- External service calls must never be in the critical synchronous path if avoidable.
- External service outage must degrade gracefully — core functionality must survive.

### 1.3 Graceful Degradation

- External dependency failure disables only dependent features.
- The system must never crash due to external failure.
- All degradation states must be reflected in health endpoints and monitoring.
- Every failure must log.

- One branch per task. Never commit directly to `main`.
- Naming: `{author}/{epic-prefix}-{task-id}-{short-description}`

### 2.2 Commits

Format: `{TASK-ID}: Imperative description`

Rules:
- Atomic commits only.
- No AI mentions anywhere. No "Generated with X". No co-author lines.
- No secrets. Imperative mood.

### 2.3 Commit / Push / PR / Merge Permissions

**Commit Permission (Per-Task Model)**

- Ask once per task branch for permission to commit locally.
- After approval, multiple local commits are allowed without re-asking.
- Permission applies only to the current branch. A new branch requires new permission.

**Push / PR / Merge**

- Explicitly ask before: `git push`, creating a PR, merging a PR.
- Never merge without explicit confirmation.
- Always merge with `--squash --delete-branch`.
- **Run tests locally before every push.** Use `docker compose up -d postgres redis` and run the full test suite against the local stack. Never push if local tests fail.
- **Wait for CI to go green before proposing a merge.** After pushing/creating a PR, check CI status and confirm the run passes. Never say "all tests pass" based on local results alone.

## 3. Code Standards

### 3.1 Rust

- Edition 2021+
- tokio multi-thread runtime
- No `.unwrap()` in production; `anyhow` for app errors, `thiserror` for libraries
- Structured JSON logs via `tracing`
- `cargo fmt` + `cargo clippy -- -D warnings`
- Exact dependency pinning
- Unit tests for all public functions
- No unbounded channels in hot path
- Profile before optimizing; avoid heap allocations in hot path

### 3.2 Python

- Python 3.11+
- Type hints required; `from __future__ import annotations` for forward references
- No bare `except`; structured logging with `structlog`
- No ORM for databases; parameterized SQL only
- No blocking calls inside async; pin dependencies
- All public functions must have unit tests
- Batch DB writes only; no unbounded in-memory buffers

### 3.3 React / TypeScript

- Strict TypeScript; no `any` unless justified with comment
- Zustand for global state; Tailwind only
- Single WebSocket connection
- Prettier + ESLint required

### 3.4 SQL Rules

- UPPERCASE SQL keywords
- Idempotent migrations (`IF NOT EXISTS`)
- No `SELECT *`; always parameterized queries
- Every time-range query must include `WHERE time` bounds
- Verify performance with `EXPLAIN ANALYZE`

### 3.5 YAML Rules

- Every parameter commented
- No anchors (`&`/`*`)
- Grouped by domain
- Config additions require schema update

## 4. Performance & Hot Path Mandate

These are defaults, not suggestions.

- One pass over collections only.
- No append-in-loop if extend/generator works.
- Use single multi-row INSERT, never `executemany`.
- No loop-await — use `gather`/`join`.
- Hoist all loop invariants.
- No rebuilding constants inside functions.
- Batch I/O.
- Profile before exotic optimization.

Before submitting PR touching data processing, verify:
- No double iteration.
- No loop-await.
- No per-row DB writes.
- No per-call constant rebuild.

## 16. Performance Reviewer Checklist (Mandatory for Data Path Changes)

Complete before approving any PR touching: data ingestion, processing pipelines, DB write paths, cache hot paths, API response paths.

If any item fails, the PR must not merge until resolved or explicitly justified.

### 16.1 Iteration Discipline

- [ ] No collection iterated more than once when a single pass would suffice
- [ ] No `append` inside Python loops when `extend(generator)` is possible
- [ ] No per-element push in Rust when `.map().collect()` or `.fold()` would work
- [ ] No nested loops creating accidental O(n²) behavior
- [ ] Loop-invariant values hoisted outside the loop

### 16.2 Async & Concurrency

- [ ] No `await` inside loops when `asyncio.gather()` can be used
- [ ] No sequential fan-out for independent I/O calls
- [ ] No blocking calls inside async functions
- [ ] Concurrency limits enforced for rate-limited APIs
- [ ] Rust async tasks use bounded channels

### 16.3 Database Write Path

- [ ] No `executemany`; no per-row `execute` calls
- [ ] Multi-row `INSERT VALUES (...), (...), ...` used
- [ ] Placeholder strings pre-built at module load
- [ ] Flat parameter list built in one comprehension
- [ ] Writes batched with both size cap and timeout flush

### 16.4 Memory Discipline

- [ ] No unbounded in-memory accumulation
- [ ] Buffers have both max size and timeout
- [ ] Large temporary lists avoided when generator suffices
- [ ] No repeated JSON serialization/deserialization in hot loops

### 16.5 Allocation & Constant Hoisting

- [ ] No SQL placeholder string rebuilt per call
- [ ] No regex compiled inside function
- [ ] No static tuple/list rebuilt inside function
- [ ] No repeated format string construction inside loop

### 16.6 Logging Performance

- [ ] No heavy debug logging inside hot loops
- [ ] Log fields are structured, not string-concatenated
- [ ] No large payload dumps at INFO level
- [ ] Debug logs disabled by default in production

### 16.7 Cache Discipline

- [ ] No cache write without TTL
- [ ] No repeated writes of identical values inside same cycle
- [ ] No N+1 cache reads inside loops
- [ ] No blocking cache calls in async hot path

### 16.8 Query Performance

- [ ] All time-range queries include explicit time bounds
- [ ] Supporting index exists for query pattern
- [ ] `EXPLAIN ANALYZE` verified for new queries
- [ ] No full table scans on large tables
- [ ] Aggregates/views used instead of raw aggregation where possible

### 16.9 Determinism & Reproducibility

- [ ] No randomness introduced
- [ ] No reliance on dict iteration order where order matters
- [ ] No floating-point instability without rounding policy
- [ ] Thresholds read from config, not embedded

### 16.10 Pre-Merge Performance Questions (Must Be Answered)

Before merging, explicitly answer:
1. Does any loop iterate the same collection twice?
2. Does any I/O fan-out run sequentially when it could be concurrent?
3. Are all DB writes batched into single multi-row inserts?
4. Are all loop invariants hoisted?
5. Could this code run 10x more frequently without collapsing?
6. Did we profile if this is in the hot path?

If the answer to any is "no", justify in PR description.

### 16.11 When Profiling Is Required

Mandatory when:
- Adding new data processing
- Modifying core evaluation logic
- Changing DB write structure
- Increasing batch sizes
- Adding cross-service computation
- Introducing new fan-out calls

Tools: `py-spy`, `cProfile` (Python); `cargo flamegraph`, `perf` (Rust).

### 16.12 Golden Rule

Performance fixes must:
- Preserve determinism
- Preserve correctness
- Preserve test coverage

Speed is never allowed to compromise correctness.

## 18. External API Quirks

### 18.1 Beds24

- **Auth endpoint**: `GET /authentication/token` with header `refreshToken: <token>`. NOT `GET /authentication/setup` (one-time invite code only).
- **Token rotation**: Happens occasionally, not on every call. `authenticate()` returns `None` when no rotation — no DB write needed.
- **Messages endpoint**: `GET /bookings/messages` (paginated). Field names: `bookingId`, `propertyId`, `message`, `source`. NOT `bookId`, `propId`, `msg`.
- **Channel field unreliable**: `booking.channel` drops to `'direct'` or empty for Booking.com bookings that go through Stripe payment (all SXM bookings). Do NOT filter messages by channel. Ingest all guest messages and use `channel` as a display label only.
- **`POST /bookings/messages` works for any `bookingId`** regardless of channel value — reply routing is always via `bookingId`. **Body must be a JSON array**: `[{"bookingId": 12345, "message": "..."}]` — sending a plain object returns `400 "Request body must be an array"`.
- **Pre-bookings**: Unconfirmed bookings are absent from `GET /bookings` but present in `GET /bookings/messages`. Default platform to `'booking'` for missing bookings — replies still work.
- **Content-based platform detection**: When a booking is absent from `GET /bookings` (archived/pre-booking), detect Airbnb by checking `'muscache.com' in body`. Fall back to `'booking'`. The conversation upsert also upgrades `booking/direct` → `airbnb` atomically if a later poll detects the better signal.
- **Beds24 message ID ordering quirk**: For archived/old bookings, all messages may share the same `sent_at` timestamp (Beds24 didn't track per-message time). Within a tied `sent_at`, lower `beds24_message_id` = *newer* message (IDs are assigned globally, not per-booking, and old bookings get high IDs relative to newer messages). Sort: `ORDER BY sent_at ASC, beds24_message_id DESC NULLS LAST`.

## 19. Media & Attachments

- **Airbnb image URLs expire in ~1 hour** (pre-signed AWS S3). Always cache at ingest time — never store raw S3 URLs in the DB.
- **Cache pattern**: `_cache_images(body)` in `ingest.py` — extract `<img src>` via regex, download via `httpx`, save to `media/attachments/{sha256}{ext}`, replace URL with `/media/attachments/filename`.
- **Serving**: `StaticFiles` mounted at `/media` in `main.py`. Files served at `http://host/media/attachments/filename`.
- **Cleanup**: `_attachment_cleanup_worker()` runs daily, deletes files older than 14 days.
- **Docker**: `media_attachments` named volume required — without it, cached images are lost on container restart.
- **Frontend fallback**: Render images with `onError` handler showing "Image expired" placeholder (catches cases where ingest missed the URL or file was cleaned up).

## 20. Conversation Deduplication Rules

- **Unique constraint**: `(guest_contact) WHERE guest_contact IS NOT NULL` — one row per contact, regardless of platform. Migration 0007.
- **Platform upgrade**: Upsert uses `CASE WHEN EXCLUDED.platform NOT IN ('booking','direct') THEN EXCLUDED.platform ELSE conversations.platform END` — ensures booking/direct is always overridden by a stronger signal (airbnb, whatsapp).
- **Guest name preservation**: Upsert uses `CASE WHEN EXCLUDED.guest_name != 'Unknown' AND EXCLUDED.guest_name != '' THEN EXCLUDED.guest_name ELSE conversations.guest_name END` — existing non-Unknown names are never overwritten.

## 22. WhatsApp ↔ Booking Cross-Platform Linking (SOLO-138)

- **Linking mechanism**: Conversations are linked via `guest_phone` column (migration 0008). WhatsApp stores E.164-without-plus as both `guest_contact` and `guest_phone`. Beds24 ingestion normalises `booking['phone'] or booking['mobile']` to digits-only and stores in `guest_phone`.
- **Separate conversations**: WhatsApp ingest always creates a **dedicated WhatsApp conversation** — it is never merged into the booking conversation. The two are linked logically, not physically.
- **Property inheritance**: `ingest_whatsapp_message` copies `property_id` from the linked booking conversation so the property name appears in the WhatsApp thread header.
- **Inbox hiding**: Linked WhatsApp conversations are filtered out of the inbox list (`_build_where` in `conversations.py`) — they are only accessible via the toggle button in the booking thread. Unlinked WhatsApp convs (no matching booking) remain visible.
- **Unread routing**: A new WhatsApp message increments `unread_count` on **both** the WhatsApp conv and the linked booking conv. WS push fires on the booking conv ID so the inbox badge updates.
- **API fields**:
  - `GET /conversations/{id}` → `linked_conversation_id`, `linked_conversation_unread` (from the linked conv's `unread_count`)
  - `GET /conversations/` → `linked_whatsapp_unread` per row (correlated subquery; `NULL` = no linked WA conv)
- **Read-state semantics**: Opening the booking conv marks it as read (badge drops to WhatsApp-only count). The WhatsApp portion clears only when the user clicks the toggle button (`switchToLinked` marks both convs as read).
- **Toggle button badge**: Red dot on the platform toggle showing `linked_conversation_unread`. Only shown when > 0.
- **Inbox badge**: `ConversationItem` displays `unread_count + (linked_whatsapp_unread ?? 0)` and a green **WhatsApp** pill when `linked_whatsapp_unread !== null`.

## 23. Local Dev — Docker File Watching

- **Vite HMR on macOS + Docker**: inotify events don't propagate through Docker volume mounts on macOS. `vite.config.ts` must have `server.watch.usePolling: true` for hot-reload to work when `./frontend/src` is mounted as a volume.
- **Backend Python changes**: The backend service uses a built Docker image (`docker compose build backend`). Python source changes require a rebuild + restart — restarting the container alone is not enough.
- **Frontend changes**: With `usePolling: true`, Vite picks up changes automatically via the `./frontend/src:/app/src` volume mount. No rebuild needed.
- **Test HMAC secret**: When curling the WhatsApp webhook locally, use the value of `WHATSAPP_APP_SECRET` from `.env` as the HMAC key, not a hardcoded `"secret"`.
